# Backend Services Conventions

## 공통
- Runtime: Bun — `bun:test` 전용 (jest/vitest 금지)
- Import: `@teleprompter/protocol` (서버용 full export)
- Logger: `createLogger("<Module>")` — 모듈명 prefix
- Error handling: log with logger, expected failures는 throw 대신 error state 반환
- Tests: 소스 옆 co-located (`foo.test.ts`)

## Daemon
- Store: append-only Record 저장 (`sessions.sqlite`)
- **`loadPushTokens()` 는 corrupt row 를 drop 만 하지 않고 PURGE 한다**: 가드(`frontend_id`/`daemon_id`/`sealed` 비어있음 또는 `platform` 이 ios/android 아님)를 통과 못한 row 는 in-memory skip + `DELETE FROM push_tokens WHERE frontend_id=?` 로 테이블에서 제거. 안 그러면 영구 unusable row 가 매 daemon 시작마다 재-read + 재-warn 돼 무한 누적된다 (실측: 옛 코드 경로가 `buildEvents` 의 `daemonId=""` 기본값을 `savePushToken` 으로 흘려 빈 `daemon_id` row 4개 생성 → 140+ "dropped corrupt push_token row" 로그). 가드 통과 row 는 그대로 유지(valid row preservation). write-side 원인(빈 daemonId)은 이미 call-site 에서 daemonId 명시로 수정됨 (`pairing-orchestrator.test.ts` "push-token regression" 가드 존재) — 이 purge 는 이미 디스크에 쌓인 stale row 의 self-heal. 회귀 가드: `store.test.ts` ("empty daemon_id row is dropped AND purged, valid rows preserved" + platform/sealed corrupt purge, raw `COUNT(*)` 로 purge 검증, source-only revert 로 genuine 입증).
- Session: SessionManager가 Runner spawn/kill 관리 (= mux: 세션당 1 Runner)
- **`handleBye` generation guard (restart race)**: `session.restart` 는 동기적으로 old Runner 를 SIGTERM(→`stop()` 이 bye 송신)하고 같은 sid 로 new Runner 를 spawn 한다. old Runner 의 bye 가 new generation 의 hello 가 등록된 *뒤* 도착하면 (minority race — old runner 는 열린 소켓으로 near-instant flush, new runner 는 cold-start 라 보통은 old-bye-가-먼저지만 역전 가능), 가드 없는 `handleBye` 는 live 세션을 "error" 로 flip + 새로 등록된 Runner 를 `unregisterRunner` 해 **실행 중 PTY 를 phantom-stopped 세션으로 고아화**한다. 그래서 Runner 는 bye 에 자기 `process.pid` 를 실어 보내고(`IpcBye.pid`, optional — old Runner 는 생략, `ipc-guard.ts` bye arm 이 present 시 positive-int 검증), `handleBye` 는 `getRunner(sid).pid !== msg.pid` 면 stale 로 보고 무시한다. `SessionManager` 의 `proc.exited` generation guard(Subprocess 참조 비교)와 동일 패턴 — IPC 레이어엔 Subprocess 핸들이 없어 pid 로 비교. 회귀 가드: `command-dispatcher.test.ts` ("stale bye from the old runner does not corrupt a restarted session" + "matching bye ... processed normally", source-only neuter → stale-bye fail 로 genuine 입증) + `ipc-guard.test.ts` (optional bye pid present/absent/invalid).
- IPC: Unix domain socket, framed JSON. Runner↔Daemon io/event/meta + CLI↔Daemon 명령 (`pair.begin`, `pair.remove`, `pair.rename`, 등). 네이티브 Windows는 미지원 — Windows 사용자는 WSL 안에서 Linux 빌드를 실행한다.
- Vault key storage: plaintext BLOBs in SQLite — filesystem 권한으로 보호
- Worktree: `git worktree add/remove/list` 직접 관리 (외부 도구 의존 없음). **`add`/`remove` 둘 다 `validatePathContainment` 로 worktree base(=`dirname(repoRoot)`) 밖 경로를 거부**하고 (trust-boundary 대칭 — 등록된 worktree 라도 base 밖이면 relay 경로로 제거 불가), git argv 에 `--` 구분자를 넣어 `-` 로 시작하는 경로가 flag 로 파싱되지 않게 한다. `add` 는 `path` 를 `resolve(repoRoot, path)` 로 절대화해 `rev-parse HEAD` cwd 와 반환 `wt.path` 에 쓴다 — 상대 경로를 cwd 로 넘기면 launchd/systemd(daemon CWD=`/`·`~`)에서 ENOENT 로 worktree.create 가 ghost dir 를 남기고 실패. 회귀 가드: `worktree-manager.test.ts`.
- Relay outbound client: `RelayConnectionManager`가 pairing당 1 `RelayClient` 유지. **Daemon은 relay의 유일한 클라이언트** (invariant) — CLI는 직접 relay WS를 열지 않음.
- Pair ops handler (`IpcCommandDispatcher`): `pair.remove` → `RelayConnectionManager.removePairing` (peer에 `control.unpair` 통지 → **store.deletePairing 먼저, 그 다음 client.dispose+splice**). store-first 순서가 핵심: dispose 후 deletePairing 이 throw 하면 client 는 사라졌는데 pairings row 가 남아 `reconnectSaved()` 가 재시작마다 부활시킨다 (store-first 면 throw 시 client 가 pool 에 남고 예외가 깨끗이 전파). `pair.rename` → `RelayConnectionManager.renamePairing` (store.updatePairingLabel → 연결된 peer에 `control.rename`). 양쪽 다 notified peer 수를 ok 응답에 반환. **동시 removePairing 가드**: `removingDaemonIds` Set 으로 같은 daemonId 의 in-flight 제거가 겹치면(inbound `control.unpair` ↔ `tp pair delete`) 두 번째 호출은 client teardown 을 건너뛰고 idempotent store delete 만 — 안 그러면 두 번째의 `sendUnpairNotice` 가 이미 닫힌 소켓에서 돌아 notified:0 을 잘못 보고. 회귀 가드: `relay-manager.test.ts`.

- **Auto-cleanup 는 throw 가 daemon 을 죽이지 못하게 가드**: `daemon.ts` `startAutoCleanup` 의 startup prune 과 24h `setInterval` body 둘 다 try/catch+log. `pruneOldSessions`→`deleteSession`→`unlinkRetry` 는 ENOENT/EBUSY/EPERM 외 errno(EACCES·EROFS) 나 SQLite 에러에 re-throw 하는데, timer 콜백 밖으로 나간 throw 는 Bun 프로세스를 종료시킨다 — long-running daemon 이 일시적 FS 조건에 통째로 죽지 않게 swallow-and-log. 회귀 가드: `auto-cleanup.test.ts`.
- **세션 DB 삭제는 WAL/SHM sidecar 까지 unlink + 기존 orphan 은 startup 에서 self-heal**: 세션 DB 는 WAL 모드(`schema.ts` PRAGMAS)라 `${sid}.sqlite-wal`/`${sid}.sqlite-shm` 두 sidecar 를 동반한다. `db.close()` 는 exclusive lock + 깨끗한 checkpoint 가 가능할 때만 sidecar 를 지우므로 실전에선 자주 남는다 — `deleteSession` 이 main `.sqlite` 만 unlink 하면 삭제/prune 마다 -wal + -shm 가 영구 누적된다(실측: 옛 빌드가 dogfood store 에 107세션×2=214 orphan 을 남김). 그래서 `store.ts deleteSession` 은 세 파일(`base`/`-wal`/`-shm`)을 모두 `unlinkRetry`(ENOENT=성공) 한다. 이미 디스크에 쌓인 orphan 은 `store.ts sweepOrphanedSidecars()` 가 startup auto-cleanup 의 try-block 에서 한 번 쓸어 self-heal — base `.sqlite` 가 없고 **열린 세션(`sessionDbs`)도 아닌** -wal/-shm 만 unlink(라이브 WAL 제거 금지 — un-checkpointed write 유실). push-token PURGE self-heal 과 동일 패턴(write-side fix + 기존 stale 자가치유). 회귀 가드: `store-cleanup.test.ts` ("deleteSession removes the WAL/SHM sidecars", "sweepOrphanedSidecars removes orphaned ... spares live + base-present") + `auto-cleanup.test.ts` ("startAutoCleanup sweeps orphaned ... on startup"; source-only revert → 2 test fail 로 genuine 입증).
- **RelayClient dispose-race 가드 (timer/socket 누수)**: `startPing()` 와 `connect()`(deriveKxKey await 직후) 둘 다 `if (this.disposed) return;` 재확인. (a) `auth.ok` 핸들러가 `broadcastDaemonPublicKey()` 를 await 하는 동안 dispose 가 끼면 stale continuation 이 `startPing` 으로 영영 안 치워지는 30s interval 을 설치 → 가드로 차단. (b) pairing 중 Ctrl+C 가 `deriveKxKey` await 창에 dispose 를 걸면 재확인 없이 nobody-owns WebSocket(phantom)이 열려 relay slot 을 idle-timeout 까지 점유 → 가드로 차단. **`scheduleReconnect` 의 `setTimeout(() => this.connect(), delay)` 는 반드시 `.catch()`** 로 감싼다 — connect() 가 reject(corrupt relayUrl·sodium-init 실패)하면 reconnect 루프가 영구 중단되고 rejection 이 logger 가 아닌 process unhandledRejection 으로 샌다. catch 에서 log+reschedule(backoff 유지). 회귀 가드: `relay-client.test.ts`.
- **`send()` 는 실제 transmit 여부를 boolean 으로 반환하고, delivery-count 호출자는 그걸 honour 한다**: `RelayClient.send()` 는 `ws?.readyState === OPEN` 일 때만 전송하고 그 결과를 `boolean` 으로 반환한다 (null/closing 소켓이면 조용히 no-op → `false`). `sendEncrypted`/`sendControl` 이 이를 전파해 `sendRenameNotice`/`sendUnpairNotice` 가 **실제로 안 나간 프레임을 notified 로 세지 않게** 한다. 핵심 레이스: 동시 `removePairing()→dispose()` 가 `cleanup()` 으로 `this.ws` 를 null 로 만들지만 (의도적으로) `peers`·`authenticated` 는 유지한다 — 그 창에서 `renamePairing` 루프가 돌면 peer 도 찾고 encrypt 도 성공하지만 `send()` 가 null 소켓에서 no-op → 고치기 전엔 `sendControl` 이 `true` 를 반환해 notified count 가 부풀고, ok 응답이 받지도 못한 peer 를 "통지됨"으로 보고했다. 회귀 가드: `relay-client.test.ts` "sendRenameNotice returns false ... when the socket was disposed mid-flight".
- **`broadcastEncrypted` 는 peer 별 best-effort (한 peer 실패가 fan-out 을 중단시키지 않는다)**: `publishRecord`/`publishState` → `broadcastEncrypted` 는 각 peer 의 `sendEncrypted` 를 try/catch 로 감싼다. N:N 라 corrupt/rotated session key 를 가진 한 peer 의 `encrypt()` throw 가 bare loop 를 abort 시키면 **iteration 순서상 그 뒤의 모든 healthy peer 가 그 record/state 프레임을 조용히 놓친다**. throw 를 warn 으로 가두고 다음 peer 로 진행 — `handleFrame` fallback 루프의 per-peer 가둠과 동일 idiom. 회귀 가드: `relay-client.test.ts` "broadcastEncrypted keeps fanning out after one peer's encrypt() throws".
- **CLI-path session.delete/prune 는 relay unsubscribe 도 한다**: `command-dispatcher.ts` 의 IPC `handleSessionDelete`/`handleSessionPrune` 은 store row 삭제 후 `getRelayClients().unsubscribe(sid)` 를 호출 — relay-plane twin(`:606`)과 동일하게. 안 하면 각 RelayClient 의 `subscribedSessions` Set 에 stale 항목이 쌓인다(bounded; data leak 아님, IPC↔relay-plane 발산만). `worktree.create` 핸들러도 `session.create`(:525-527)처럼 createSession 직후 즉시 `subscribe(sid)` — 안 하면 runner hello round-trip 전 첫 입력 프레임이 relay 에서 drop. 회귀 가드: `command-dispatcher.test.ts`.
- **Pairing cancel-during-connect 는 깨끗한 에러**: `pending-pairing.ts` `begin()` 은 `await relay.connect()` 후 `settled` 재확인 → cancel 이 `this.relay` 를 null 로 만든 뒤의 `subscribe()` null-deref("relay unreachable: Cannot read properties of null") 대신 `"pairing cancelled during relay connect"` throw. 회귀 가드: `pending-pairing.test.ts`.

### Daemon robustness guards (daemon-audit)

5 건의 autonomous-safe 강건성 가드 — 각각 *source-only revert → 테스트 fail* 로 genuine 임을 증명함:

- **rank 1 — `onFrontendJoined` delegate throw 는 슬롯을 wedge 하지 않는다**: `pairing-orchestrator.ts` 의 production `createRelayClient` (non-factory) 경로는 `events.onFrontendJoined?.(frontendId)` 를 try/catch 로 감싸고 *그 다음* `pp.__markCompleted(frontendId)` 를 무조건 실행한다. delegate 는 `RelayConnectionManager.buildEvents` 의 side-effects (예: `store.listSessions()` SQLite 쿼리) 를 돌리는데 transient DB/FS 에러로 throw 할 수 있다 — 안 가두면 단일-슬롯 PendingPairing 이 영영 resolve 안 되고 (pairing 타임아웃 없음) 이후 모든 `begin()` 이 `already-pending` 으로 던져 `tp pair new` 가 daemon 수명 내내 wedge 된다. frontend 는 이미 kx 완료라 pairing 은 *완료된 것* — hello/subscribe 딸꾹질이 슬롯을 strand 하면 안 됨. 회귀 가드: `pairing-orchestrator.test.ts` ("rank-1 onFrontendJoined guard"; factory 경로를 우회해 `mock.module` 로 inert `RelayClient` 를 stub 하고 wrappedEvents 를 캡처).
- **rank 2 — `onMessage` throw 는 OUR 가드가 가둔다 (Bun error fallback 아님)**: `ipc/server.ts` Bun socket `data` 콜백은 `self.events.onMessage(...)` 를 try/catch 로 감싸 (decode-error 경로와 동일 idiom) attributable `"onMessage handler threw"` 로그 + 해당 소켓만 `end()`. `onMessage`→`dispatchIpc`→`handleRec/Hello/Bye` 는 동기 SQLite write 라 disk-full·SQLITE_BUSY·corrupt page 에 throw 가능. **현 Bun 빌드에선 가드 없어도 escaped throw 가 Bun 의 socket `error` 핸들러(server.ts:119)로 routing 돼 daemon 이 *생존*하지만**, 그건 undocumented·version-dependent fallback — 가드는 containment 를 우리 결정론적 코드 경로로 만든다. 그래서 회귀 테스트는 *survival* 이 아니라 **로그 attribution** 을 assert 한다 (`console.error` spy: `"onMessage handler threw"` 존재 + `"socket error:"` 부재 — survival 만으론 fix-sensitive 하지 않음). 회귀 가드: `ipc/server.test.ts` ("rank 2").
- **rank 3 — frontend-supplied `sid` 는 path-traversal 가드를 통과해야 한다**: `assertSafeSid(sid)` (allowlist `/^[A-Za-z0-9_-]+$/`, `packages/protocol/src/socket-path.ts`, index+client export) 가 `command-dispatcher.ts` `session.create` arm 의 try 블록 *맨 앞* + `store.ts` `createSession` 맨 앞 (defense-in-depth) 에서 검증. relay 로 들어온 `msg.sid` 는 Store 의 `join(storeDir,'sessions',sid+'.sqlite')` 로 흘러가 crafted `../../evil` 가 daemon-writable 임의 경로에 SQLite 파일 create/unlink + (runner-crash 가 never cleanup 하는) relay subscription leak 을 유발. 가드는 createSession/subscribe *전* 에 거부 → 부작용 0 의 clean `SESSION_ERROR`. 자동생성 (`session-<base36ts>`) · worktree (`<sanitizeForSid(branch)>-<ts>`) sid 는 항상 통과. 회귀 가드: `socket-path.test.ts` (`assertSafeSid` 단위) + `command-dispatcher.test.ts` (traversal sid → `createSession` 호출 0 + subscribe 0 + `SESSION_ERROR` 통합).
  - **worktree sid 는 branch 를 `sanitizeForSid` 로 통과시킨다 (orphan 방지)**: `command-dispatcher.ts` `handleRelayWorktreeCreate` 는 worktree sid/기본 경로를 `<sanitizeForSid(branch)>-<ts>` 로 짓는다. 옛 derivation 은 `branch.replace(/\//g,"-")` 로 `/` 만 평탄화했는데, `git check-ref-format --branch` 는 sid allowlist(`[A-Za-z0-9_-]+`)보다 훨씬 관대해 `.`(`release-1.2`·`v2.0`)·`+`·non-ASCII 글자를 허용한다 — 그런 legal branch 의 sid 는 `store.createSession` 의 `assertSafeSid` 를 통과 못해 throw 하는데, **이미 `wm.add` 가 디스크에 worktree 를 만든 *뒤*** 라 worktree 가 고아가 되고 혼란스러운 "invalid sid" 에러만 남았다. `sanitizeForSid`(`packages/protocol/src/socket-path.ts`)는 모든 non-allowlist char 를 `-` 로 접고 run 을 합치고 양끝 `-` 를 trim 해 sid 가 항상 allowlist-clean 이게 한다(전부 non-allowlist 인 branch 는 `"wt"` 로 fallback). 원본 branch 는 git 에 verbatim 전달 — sanitize 는 local sid/기본 worktree dir 이름에만 적용(no wire/schema/peer impact). 회귀 가드: `socket-path.test.ts` (`sanitizeForSid` 단위 + output-always-passes-`assertSafeSid` 불변식) + `command-dispatcher.test.ts` ("worktree.create sanitizes a '.'-containing branch ..." — `release-1.2` → sid allowlist-clean + createSession 도달, source-only revert 로 genuine 입증).
  - **worktree.create 는 `createSession` 실패 시 worktree 를 롤백한다 (orphan 방지)**: sid sanitization 으로 `assertSafeSid` 경로는 닫혔지만, `wm.add` 가 디스크 worktree 를 만든 *뒤* `createSession`(동기 SQLite write + per-session DB open)이 transient 실패(disk-full·SQLITE_BUSY·corrupt page·base36-ts sid 충돌)로 여전히 throw 할 수 있다. 그러면 `withWorktreeManager` catch 가 `WORKTREE_ERROR` 만 보내고 session-less worktree 가 디스크에 고아로 남는다 (documented `deleteSession` 비원자성의 analog). `handleRelayWorktreeCreate` 는 `createSession` 을 try/catch 로 감싸 throw 시 **best-effort `wm.remove(wt.path, true)`** 로 worktree 를 롤백한다 (`force:true` — 우리가 방금 만든, 커밋 없는 worktree). 롤백은 원본 실패를 **절대 가리지 않는다**: `wm.remove` 자신이 throw 할 수 있어 독립 try/catch 로 가두고 warn 만 남긴 뒤 *원본* `createSession` 에러를 re-throw 해 사용자 frame 에 싣는다 (pairing-orchestrator 의 orphan-relay best-effort dispose 와 동일 idiom). success-path 동작·사용자에게 가는 에러는 불변 — 실패 경로에 cleanup 만 추가. 회귀 가드: `command-dispatcher.test.ts` ("worktree.create rolls back the worktree when createSession throws" — `removeCalls` 1회·force=true·path 일치·`WORKTREE_ERROR`+원본 메시지·worktree.created 없음 + "surfaces the original error even if rollback also fails" — remove 가 throw 해도 원본 에러가 이김; source-only revert → 2 test fail 로 genuine 입증).
  - **worktree.remove 는 live-session 가드를 친다 (refuse-non-force, kill-on-force)**: `git worktree remove` 는 worktree 안에 cwd 를 둔 *running* 세션을 보호하지 않는다 — clean worktree 면 non-force remove 가 살아있는 프로세스가 안에 있어도 성공하고, POSIX 에선 그 프로세스가 unlinked cwd 로 계속 돌아 세션이 조용히 디렉터리를 잃는다 (kernel 이 inode 를 hold). `handleRelayWorktreeRemove` 는 `wm.remove` *전에* 가드한다: target path 를 `wm.canonicalize`(=`validatePathContainment` 와 동일한 resolve+realpathSync-parent+lexical-fallback; leaf 는 realpath 안 함 — 저장값이 `add` 의 `resolve(repoRoot,path)` 라 leaf 를 풀면 발산) 로 정규화하고, **truth source = `sessionManager.listRunners()`(live 런너 맵)** 로 worktree 를 참조하는 세션을 찾는다 (store 의 `state` 컬럼 아님 — row 가 stale 'running' 일 수 있어 거기 blocking 하면 정당한 remove 를 오거부). 각 런너의 worktree 경로 = `runner.worktreePath ?? store.getSession(sid).worktree_path`(hello 경로로 worktreePath 없이 등록된 old-protocol 런너 fallback). **non-force + blocker** → blocking sid 목록을 실은 `WORKTREE_ERROR` 를 publish 하고 **`return`(throw 아님 — `withWorktreeManager` 가 두 번째 에러 frame 을 안 내게)**, `wm.remove` 미호출. **force + blocker** → process 핸들 없는 blocker(passthrough/registered-only — `killRunner` 가 no-op 이라 죽일 수 없는데 dir 만 지우면 live 프로세스 위에서 remove)가 있으면 그것도 `WORKTREE_ERROR` 로 거부; 아니면 각 blocker 를 `killRunner → **await waitForExit** → unregisterRunner → updateSessionState("stopped")` (handleSessionDelete 미러 + **process exit 을 await** — `killRunner` 는 SIGTERM 만 보내므로 안 기다리면 git 이 살아있는 PTY 의 cwd 와 race). kill 루프 중 throw → re-throw 로 `WORKTREE_ERROR` 표면화 + `wm.remove` 미도달(안전 방향: worktree 디스크 보존). unsubscribe 는 kill 루프 **밖**(handleSessionDelete 처럼) → 부분 실패가 relay 구독을 stranded 시키지 않게. `SessionManager.waitForExit(sid)` 는 추적된 `Subprocess.exited` 를 await(없으면 즉시 resolve). 회귀 가드: `command-dispatcher.test.ts` (10 케이스 — non-force refuse/proceed/stale-row, force kill-order(waitForExit 포함)·passthrough-refuse·stored-path-fallback·trailing-slash canonical·kill-failure containment; source-only revert → 6 test fail) + `session-manager.test.ts` (`waitForExit` unknown/registered-only/killed-process 3 케이스).
- **rank 4 — `session.export` truncated off-by-one**: handler 는 `effectiveLimit + 1` 행을 fetch 하고 `fetched.length > effectiveLimit` 일 때만 truncated, 그 경우 `effectiveLimit` 으로 slice. 옛 `records.length >= effectiveLimit` 는 *정확히* limit 행일 때 false-positive — 전체 히스토리가 다 돌아와도 truncated:true 로 사용자에게 가짜 "잘림" 경고를 보였다. 회귀 가드: `command-dispatcher.test.ts` ("rank 4"; exactly-limit → truncated:false + `effectiveLimit+1` fetch 요청 assert, more-than-limit → truncated:true + slice).

## Runner
- PTY: `Bun.spawn({ terminal })` — ANSI 출력 수집
- Hooks: Claude Code hooks events 수집 → Record 생성 → IPC 전송
- Settings: 기존 Claude Code settings와 merge (`settings-builder`)
- **IPC decode-throw teardown 불변식**: `IpcClient` 의 `data` 콜백은 `FrameDecoder.decode()` 를 **반드시 try/catch** 로 감싼다. decode 는 protocol-fatal 프레임(oversized H1 / malformed-JSON M1, `codec.ts`)에서 throw 하는데, Bun 은 `data` 콜백 밖으로 나간 throw 를 socket `error`/`close` 로 변환하지 **않는다** → guard 없으면 socket 이 죽은 stream 위에서 `{connected:true}` 로 남아 `send()` 가 모든 PTY io/hook 이벤트를 조용히 drop 한다. catch 에서 `decoder.reset()` + (connected 면) `socket.end()` 로 close 핸들러를 깨워 Runner teardown 을 유발. 회귀 가드: `ipc/client.test.ts` "IpcClient decode-throw teardown".
- **hook socket sid traversal 가드**: `HookReceiver.defaultSocketPath(sid)` 는 `sid` 가 `/`·`\`·`..` 를 포함하면 join 전에 throw. `sid` 는 `--tp-sid` passthrough 플래그에서 오므로(자동생성 `session-<ts>` 외에 crafted 값 가능) per-user runtime dir 밖으로 socket 을 bind/unlink 하는 self-DoS 를 막는다. `Runner` 생성자가 이 헬퍼를 호출 → 잘못된 sid 는 생성 시점에 throw 하며, `index.ts`/`run.ts` 는 생성자 호출을 try/catch 로 감싸 clean fatal 출력.
- **hook buffer cap 은 UTF-8 바이트 기준**: `MAX_HOOK_BUF_BYTES`(1 MiB) 체크는 `Buffer.byteLength(buf, "utf-8")` 로 — JS string `.length` 는 UTF-16 code unit 이라 4-byte 코드포인트 flood 시 실제 footprint 가 cap 의 2-4배까지 자란다. (close 핸들러의 `_buf.length > 0` 비어있음 체크는 인코딩 무관하므로 `.length` 유지가 정상.)
- **graceful shutdown 은 bye flush tick 후 exit**: `index.ts`/`run.ts` 의 `gracefulShutdown` 은 `runner.stop()`(bye 프레임을 QueuedWriter 에 enqueue + `socket.end()`) 직후 `process.exit(0)` 하면 event loop 가 pending write 를 flush 하기 전에 죽어 bye 를 잃는다. `await new Promise(r => setImmediate(r))` 한 macrotask 양보 후 exit. daemon `proc.exited` 모니터가 bye 유실을 보상(세션은 결국 'stopped')하므로 cosmetic latency 만 제거하지만 clean-exit 경로를 실제로 clean 하게 만든다. 더블 시그널 force-exit(`exit(1)`) 가드는 `stopping` 불리언으로 유지.
- **`Runner.stop()` 은 PTY child 를 kill 한다 (orphan 방지)**: `stop()` 은 PTY 자신의 `onExit`(child 이미 죽음 → `kill()` no-op) 뿐 아니라 **graceful SIGTERM/SIGINT 경로**(`run.ts` → daemon `killRunner` 가 runner 프로세스에 SIGTERM → `gracefulShutdown` → `runner.stop(143)`)와 **IPC `onClose` 경로**(QueuedWriter overflow / socket teardown → `this.stop(-1)`)에서도 호출된다. 뒤 두 경우 claude child 는 아직 살아있어, `stop()` 이 `this.pty.kill()` 을 안 하면 runner 프로세스가 exit 하며 claude 를 init 으로 orphan 시켜 프로세스(+cwd/worktree hold)를 누수한다. `start()` 의 에러 cleanup 이 이미 `this.pty.kill()` 을 하므로 `stop()` 도 대칭으로 호출 — `PtyBun.kill` 은 `this.proc?.kill()` 이라 exited/unspawned proc 에 idempotent(모든 call path 안전). 회귀 가드: `runner.test.ts` "Runner.stop() kills the PTY child" (fake pty 의 kill count 0→1→1, source-only revert 로 genuine 입증).

- Stateless: ciphertext 전달만 — 복호화 불가 (zero-trust)

## Platform Gotchas (실제 반복된 함정)

- **No native Windows**: `tp` CLI exits at startup on `process.platform === "win32"` with a WSL pointer. Don't add `process.platform === "win32"` branches to source — Windows users run the Linux build inside WSL where everything is POSIX.
- **fs.rm / fs.rmdir on busy dirs**: 테스트에서 worktree/temp dir를 정리할 때 핸들이 아직 열려 있으면 일부 환경에서 `EBUSY`/`EPERM`을 던진다. `fs.rm(dir, { recursive, force })` 직접 호출 대신 `@teleprompter/protocol/test-utils`의 `rmRetry`를 쓴다 (retry with backoff, 핸들 해제 대기).
- **jsdom navigator mutation 격리**: jsdom 환경에서 전역 `navigator.*`를 건드리는 테스트는 **파일 단위로 격리**해야 한다. 같은 파일 안에서 `beforeEach`로 되돌리는 것은 다른 테스트 파일이 같은 vm context를 공유하면 오염된다. `bun:test` 기준: navigator 만지는 테스트는 별도 `*.test.ts` 파일로.
- **Unix socket path 길이**: macOS/Linux의 sockaddr_un은 path 길이가 104/108 bytes로 제한. `socket-path.ts` `resolveRuntimeDir()` 는 `XDG_RUNTIME_DIR` → `/run/user/<uid>` → `/tmp/teleprompter-<uid>` (mode-0700) 순으로 짧은 base dir 만 고르므로 정상 경로(`<base>/daemon.sock`, `<base>/hook-<sid>.sock`)는 한도 안에 든다 (hash 단축 코드는 **없다** — 짧은 base 선택으로 회피). 신규 IPC 엔드포인트는 이 헬퍼(`getSocketPath()`)를 재사용하고, base dir 를 길게 만들지 말 것.
- **IPC socket dirent self-heal (split-brain 방지)**: in-kernel listening socket 은 dirent unlink 후에도 살아있지만 path 로 `connect()` 하면 ENOENT (macOS AF_UNIX = VFS, abstract namespace 없음) — daemon 재시작 레이스 / stray `tp daemon start` 의 pre-unlink-후-lock-early-exit / OS tmp churn 으로 dirent 가 사라지면 살아있는 daemon 이 모든 신규 `tp` 클라이언트(`tp status`·passthrough `ensureDaemon`)에게 "not running" 으로 보여 **중복 daemon spawn → SQLITE_BUSY 데드락** 위험. `IpcServer` 가 `SOCKET_HEAL_INTERVAL_MS`(30s) heartbeat 로 bound path 의 dirent 를 `lstat`/`isSocket()` 체크해 사라졌으면 `server.stop()`(이미 accept 된 runner 연결은 유지) 후 같은 path 에 `Bun.listen` 재바인딩한다. timer 콜백은 try/catch (timer throw = 프로세스 종료, auto-cleanup·data-callback 가드와 동일 규율) + `unref()`. `stop()` 이 timer clear + `stopped` 플래그로 teardown 후 부활 방지. `tp doctor` 도 inline path 재구성 대신 `getSocketPath()` 경유(과거 `/run/user/<uid>` 브랜치 누락으로 stale path 출력하던 버그 수정). 회귀 가드: `ipc/server.test.ts` ("re-binds the socket after its dirent is unlinked", source-only revert 로 genuine 입증 + healthy-no-rebind / stop-no-heal 케이스).
- **`tp pair new` concurrency**: 동시 실행을 `proper-lockfile`로 막는다 (`pair.lock`). pair-related 변경 시 이 lock을 건너뛰는 코드경로를 만들지 말 것 — 동시에 두 개의 PendingPairing이 relay에 떠서 identity 꼬인다.
- **RelayClient reconnect/ping**: `packages/daemon/src/transport/relay-client.ts`의 `RECONNECT_MAX_MS` / `PING_INTERVAL_MS`를 줄이면 재연결 폭주가 날 수 있다. 수치를 바꾸기 전에 이 파일의 테스트(`relay-client.test.ts`)가 handshake/reconnect race를 재현하는 fake relay로 검증하는지 확인.
- **Dead-pairing throttle (reconnect storm 방지)**: daemon 은 시작 시 저장된 *모든* 페어링에 대해 RelayClient 를 띄운다 (`relay-manager.ts:reconnectSaved`, N:N 지원). 죽은 페어링(닫힌 탭·옛 앱 인스턴스·스캔 안 된 QR)은 socket 은 열리지만 frontend 가 영영 join 하지 않아 무한 재연결한다 — 이게 누적되면 relay 를 폭주시켜 *살아있는* 페어링까지 backfill 을 못 받는다 (실측: 9 페어링 → 41h 동안 3113 re-auth → 폰 Chat/Terminal 빈 화면). `computeReconnectPlan(attempt, peerlessReconnects)` (pure, export) 이 이를 막는다: peer 없이 `PEERLESS_RECONNECT_THRESHOLD`(3)회 재연결하면 backoff 를 `PEERLESS_RECONNECT_MS`(30분)로 throttle, `handleKxFrame` 에서 실제 frontend 가 join 하면 `peerlessReconnects=0` 으로 리셋해 즉시 fast backoff 복귀 ("폰 며칠 자다 깨어남" 정상 케이스 안 깨짐). `peerlessReconnects` 는 `ws.onopen` 에서 리셋하지 *않는다* — 죽은 페어링은 socket 은 열리므로 onopen 리셋 시 throttle 이 영영 안 걸린다. 회귀 가드: `relay-client.test.ts` "computeReconnectPlan (dead-pairing throttle)" describe.
  - **throttle 상태는 `tp doctor` 에 정직하게 노출된다 (오진 방지)**: throttle 중인 페어링은 `isConnected()===false` 이지만 이건 **정상 idle** 상태(닫힌 탭·죽은 페어링)이지 relay outage/auth 실패가 아니다. `RelayClient.isThrottled()`(`peerlessReconnects >= PEERLESS_RECONNECT_THRESHOLD`)가 이를 노출하고, `daemon.ts getRelayHealth` 가 `throttled` 를 `doctor.probe.ok` relay status 에 실어 보낸다 (`IpcDoctorRelayStatus.throttled?`, wire optional — 옛 daemon 은 생략, reader 는 absent=false). `doctor.ts checkRelayConnectivityViaIpc` 는 `connected=false && throttled` 를 **issue 로 세지 않고** "idle — no frontend connected yet (dead-pairing backoff; not an outage)" 로 표시한다 (옛 무조건 "disconnected (relay unreachable or auth failed)" 오진 수정 — throttle 중 doctor 를 돌리면 정상 idle 을 relay 장애로 잘못 보고했다). 회귀 가드: `command-dispatcher.test.ts` "doctor.probe surfaces the throttled flag for a peerless (idle) pairing" + `ipc-guard.test.ts` "doctor.probe.ok" describe (throttled present/absent-backcompat/non-boolean-reject).

- **에러 전파 (조용한 swallow 금지)**: `upgrade.ts`(`upgradeTp` catch 가 `throw err` 후 caller 가 `process.exit(1)`), `passthrough.ts`(service-daemon 경로의 `spawnRunner` try/catch → friendly hint + exit 1), `session-cleanup.tsx`(`readlinePrompt` EOF/Ctrl+D 시 hang 대신 빈 답=abort; 삭제 직전 fresh Store 로 TOCTOU 재확인해 재시작된 sid skip), `daemon.ts`(`checkDaemonLockAlive` 로 살아있는 pid 만 signal — recycled PID 오인 방지).
