# CI/CD & Deployment Conventions

## GitHub Actions
- CI: 7개 독립 병렬 job (`lint`, `type-check`, `test`, `build-cli`, `rust`, `swift-build`, `swift-smoke-ios`). 앞 5개는 ubuntu-latest, `swift-build`/`swift-smoke-ios`만 macos-26. **Bun-deletion cascade #5 PR3 이후 모든 ubuntu job body 가 Rust(cargo)** — job **이름**은 바이트 단위로 유지(required context `lint`/`type-check`/`test`/`build-cli`/`rust` — Ruleset 14604664 는 이름으로 매칭하므로 body 만 스왑하면 ruleset PATCH 0회, job-level `name:` 을 절대 추가하지 않음: required job 에 붙이면 context 가 orphan 돼 영구 pending wedge). `./.github/actions/setup`(Node/Bun/pnpm) 은 어느 ci.yml job 도 더 이상 consume 하지 않는다(setup-bun 액션 파일 자체 삭제는 PR6/PR7 — 그때 `grep 'uses: ./.github/actions/setup' ci.yml` 이 빈 결과여야 제거).
  - **`lint` job** = ubuntu-latest, `defaults.run.working-directory: rust`. `dtolnay/rust-toolchain@1.96.0`(rustfmt) + `Swatinem/rust-cache(workspaces: rust)` → `cargo fmt --all -- --check`. (PR3 이전: `pnpm exec biome ci .` — biome 가 린트하던 TS 백엔드는 PR6 에서 삭제, 살아남는 코드는 Rust 라 fmt/clippy 게이트가 SoT.)
  - **`type-check` job** = ubuntu-latest, `working-directory: rust`. `dtolnay/rust-toolchain@1.96.0` + rust-cache → `cargo check --workspace --all-targets`. (PR3 이전: 5개 `npx tsc --noEmit` — 그 TS 패키지는 PR6 삭제, Rust 타입 게이트는 컴파일러가 강제.)
  - `rust` job = ubuntu-latest, `rust/rust-toolchain.toml` 핀(1.96). 3 step: `cargo fmt --all -- --check` (rustfmt.toml: edition 2021 / max_width 100) → `cargo clippy --workspace --all-targets` → `cargo test --workspace` (TS↔Rust 골든벡터 `wire_vectors.rs`/`message_vectors.rs` 포함).
    - **Lint 심각도는 CLI 플래그가 아니라 `rust/Cargo.toml`의 `[workspace.lints]` 테이블이 SoT** — `clippy::all = "deny"`(hard gate), `clippy::pedantic = "warn"`(non-blocking, 나중에 deny 로 ratchet). 그래서 로컬 `cargo clippy`와 CI 가 바이트 단위로 일치한다. **clippy step 에 `-- -D warnings` 를 절대 붙이지 않는다** — 그 플래그는 pedantic 그룹(~125 warning)까지 error 로 over-promote 해서 빌드를 깬다(테이블이 이미 정책을 인코딩하므로 redundant + harmful). `rust.unsafe_code = "forbid"` 로 all-safe-code 불변식이 컴파일러 강제(두 crate 모두 uniffi 0.28 scaffolding 포함 clean).
  - `test` job = ubuntu-latest, `timeout-minutes: 15`, `working-directory: rust`. `dtolnay/rust-toolchain@1.96.0` + `Swatinem/rust-cache(workspaces: rust)` → `cargo test --workspace`(TS↔Rust 골든벡터 `wire_vectors.rs`/`message_vectors.rs` 포함). PR3 이전에는 `./.github/actions/setup`(bun) → `cargo build --release --bin tp-runner` + `--bin tp-daemon-probe` → `bun test --coverage`(protocol/daemon/runner/cli/relay) 였고, 그 두 cargo build 는 오직 **Bun-테스트 differential 파리티 게이트**(runner-parity/store-parity/worktree/relay-client/push/dispatcher `*-rust-parity.test.ts`)를 SKIP→RUN 으로 전환하려는 enabler 였다 — **PR4 에서 그 6개 파리티 게이트 + `tp-daemon-probe` bin 을 삭제**(Rust workspace test 가 이미 golden-vector 로 byte-exactness 를 커버; `tp-runner` 는 출하 bin 이라 유지, probe 만 제거), PR6 에서 TS 패키지 자체를 삭제하므로 body 가 순수 `cargo test --workspace` 로 축약됐다. (로컬 실 claude E2E `TP_E2E_RUNNER_BIN` 게이트와 별개 레이어 — 그건 로컬 전용, 이건 CI 결정론.)
  - `build-cli` job = ubuntu-latest, `timeout-minutes: 15`, `working-directory: rust`. `dtolnay/rust-toolchain@1.96.0` + rust-cache → `cargo build --release --bin tp` → 네이티브 smoke: `target/release/tp version`/`--help` + `tp run`→tp-runner 네이티브 exec seam(`TP_RUNNER_BIN` echo 스크립트로 `RUNNER_GOT: --sid x` grep). **아티팩트 업로드 없음**(downstream 소비자 없음 — `release.yml` 이 독립 빌드). PR3 이전: `bun run scripts/build.ts --bundle`(Rust `tp` + Bun `tpd` SEA 타르볼) + `tpd` blob seam smoke — `tpd`(출하 아티팩트의 마지막 Bun, passthrough fallback blob)는 PR6 에서 TS 백엔드와 함께 삭제.
  - `swift-build` job = **macos-26** (B4, ADR-0002). Apple 멀티플랫폼 SwiftUI 타깃이 **개발자 머신 없이도** 컴파일됨을 증명하는 결정론적 CI 게이트 — `TP_PLATFORM=macos scripts/ios.sh build` (네이티브 macOS destination, **compile-only**: Simulator 부팅·XCTest·실 claude E2E 없음, 그건 로컬 `scripts/ios.sh smoke` 담당). step: `dtolnay/rust-toolchain@1.96.0` + **9 Apple 타깃 전부** `rustup target add`(`build-xcframework.sh ensure_targets` 가 누락 시 hard-die, subset 플래그 없음 — visionOS/watchOS 포함 9개 모두 stable 1.96 Tier 2) → `Swatinem/rust-cache` → `brew install xcodegen`(러너 미제공, `cmd_gen` 가 `require xcodegen` 로 hard-die) → `scripts/ios.sh build`(= `ensure_xcframework`[9슬라이스 xcframework+UniFFI bindgen, `TP_SKIP_RUST`/`TP_FORCE_RUST` 둘 다 unset] → `ensure_project`[xcodegen generate] → `xcodebuild -destination platform=macOS`). **러너는 반드시 macos-26 핀** (bare `macos-latest` 는 macos-15→macos-26 migration 중; 앱이 iOS/macOS 26 SDK + Swift 6.0 모드, 로컬 dev = Xcode 26.5). cold rust-cache 시 ~20-25m 가능 → `timeout-minutes: 40`. **non-required** (아래 required 목록 참조) — flaky macOS 러너/SDK drift 가 무관 PR 을 wedge 하지 않게; main 에서 안정 green 확인 후 required 승격.
  - `swift-smoke-ios` job = **macos-26** (B4 후속). `swift-build` 가 *컴파일*을 증명한다면 이건 앱이 실제로 *실행*되며 전체 wire/E2EE 경로가 동작함을 증명하는 **결정론적 iOS Simulator smoke E2E**: 헤드리스 iOS Simulator 부팅 → in-process loopback relay + 가짜 daemon peer(**PR3 이후 self-contained Rust `tp-loopback` cargo bin** — 실 `tp` daemon 없음, `claude` 없음) → golden `--tp-smoke-url` 페어링 링크로 앱 런치 → unified log 에서 8마커(TP_BOOT_OK→TP_INPUT_OK) 폴링. 가짜 daemon 이 입력 probe 를 byte-echo 하므로 M5 입력 왕복도 PTY 없이 결정론적. step: `dtolnay/rust-toolchain@1.96.0`+9 타깃 → `Swatinem/rust-cache` → `brew install xcodegen` → `TP_RUST_LOOPBACK=1 TP_PLATFORM=ios TP_SIM="iPhone 17 Pro" TP_SKIP_ARM64_32=1 scripts/ios.sh smoke`(rust→gen→build→install→launch→8마커, `TP_SKIP_RUST` unset 이라 xcframework fresh build — rust-cache 가 warm). **PR3 에서 `./.github/actions/setup`(bun) 제거** — setup 은 옛 Bun loopback 스크립트(`local-relay-loopback.ts`)의 `@teleprompter/*` import 해소용이었고, `TP_RUST_LOOPBACK=1` 이 `start_loopback` 을 self-contained `tp-loopback`(PR2a) 로 플립하므로 Bun 불필요(job 의 Rust toolchain + rust-cache 가 on-demand 빌드). `sim_udid` 가 `$TP_SIM` exact-miss 시 newest iOS runtime 의 아무 iPhone 으로 fallback 하므로 러너 device lineup drift 에 강함. simctl 은 headless-by-design 이라 GUI 세션 불필요 — 그래서 macOS `open` 경로가 아닌 iOS Simulator 경로가 CI-적합. **로컬 전용 real-claude E2E(`TP_E2E_CLAUDE*`)는 절대 여기서 안 돈다** (credential-free, 결정론). cold xcframework 비용은 swift-build 와 동일 → `timeout-minutes: 40`. **non-required** (flaky Simulator boot 가 무관 PR wedge 방지; 안정 green 후 승격).
  - **아래 TS strictness / Biome 규칙은 살아있는 TS 소스(PR6 까지 존속)에 여전히 적용된다** — PR3 에서 CI `type-check`/`lint` job body 는 cargo 로 스왑됐지만, TS 소스가 남아있는 동안 로컬 lefthook pre-push(`tsc --noEmit` ×5, biome)가 이 규칙을 계속 강제한다(PR6 에서 소스+훅 동시 제거).
    - **TS strictness SoT = `packages/tsconfig/base.json`** (모든 패키지가 `bun.json`→`base.json` 체인으로 상속). 현행 켜짐: `strict`, `noUncheckedIndexedAccess`, `noPropertyAccessFromIndexSignature`, `noImplicitOverride`, `noImplicitReturns`, `noFallthroughCasesInSwitch`, **`verbatimModuleSyntax`** (T4a), **`exactOptionalPropertyTypes`** (T4c — `{x: T|undefined}` 를 `x?: T` 에 할당 금지, present-undefined vs absent 구분). 아직 안 켠 것: `noUnusedLocals`/`noUnusedParameters` (Biome `noUnusedVariables` warn 으로 커버 중). flag 추가 전 항상 5개 패키지에 `tsc --noEmit -p <pkg> --<flag>` 로 blast radius 실측.
    - **exactOptionalPropertyTypes 수정 정책 (T4c, 향후 위반 처리 규칙):** 기본은 **타입 정의 widen** (`x?: T` → `x?: T | undefined`) — behavior-preserving, TS 공식 권장. **wire 객체(`SessionMeta`/`SessionRec`/relay·ipc 메시지)에서는 절대 키를 조건부 omit 하지 말 것** — 프론트엔드/테스트가 안정적 키셋을 기대하므로 wire 타입을 widen 하고 값은 항상 present(`field: val ?? undefined`). 조건부 omit(`...(x !== undefined && {x})`)은 **non-wire constructor option-bag·React/Ink prop·`Bun.spawn` option** 에만 허용(`?? default` 로 읽혀 absent==undefined). guard(`*-guard.ts`) 검증 로직은 손대지 말 것. (회귀 사례: `toSessionMeta` 가 wire 키를 omit 해서 "all wire-format keys" 테스트가 깨짐 → widen+always-present 로 수정.)
    - **Biome lint 심각도 = `biome.json` `linter.rules` SoT** — `suspicious.noExplicitAny = "error"` (T4a; 프로덕션 코드에 `any` 0개, 테스트 2곳은 `biome-ignore` 처리됨), **`style.noNonNullAssertion = "error"`** (T4b; 110개 `!` non-null assertion 사이트를 전부 제거 — prod 는 loop/regex 불변식에 맞춘 방어적 guard-throw, test 는 local + expect 패턴. override 없이 prod/test 동일 적용). 둘 다 hard gate. `biome ci .` 는 error 에서만 non-zero exit — warn/info(예: `useLiteralKeys` 639 infos)는 CI 통과. **non-null assertion 을 새로 도입하지 말 것** — `arr[i]`/`map.get(k)` 는 `noUncheckedIndexedAccess` 로 `T | undefined`; local const + `if (x === undefined) throw`(불변식) 또는 optional chaining 으로 narrow.
- **Required status checks** (Ruleset 14604664, live): `lint`/`type-check`/`test`/`build-cli`/`rust` 5개가 필수 merge gate (`rust` 는 안정 green 확인 후 이미 승격됨). `swift-build` 와 `swift-smoke-ios` 는 아직 **non-required** — main에서 안정적으로 green 확인 후 required 목록에 추가한다 (없는 context를 required로 걸면 모든 PR이 wedge — 그래서 신규 job 은 항상 non-required 로 먼저 들어간다). **merge method = squash-only** (ruleset 강제 — 3종 merge 중 squash만 허용).
  - **PR3(#5 cascade) 가 이 5개 job body 를 Bun→Rust 로 스왑하면서 job KEY(=context 이름)는 바이트 단위로 보존했다** — GitHub 은 required context 를 job id 로 매칭하므로 이름만 유지하면 body 를 통째로 바꿔도 ruleset 을 건드릴 필요가 없다. **required job 에 job-level `name:` 을 절대 추가하지 않는다**(붙이면 context 가 job id 와 어긋나 orphan → 그 required context 가 영구 pending 으로 남아 모든 PR 이 wedge). 이름 rename 도 동일 파국이라 금지.
- Secrets: `RELAY_HOST`, `RELAY_USER`, `RELAY_SSH_KEY`, `CLAUDE_CODE_OAUTH_TOKEN`

## Release (`release.yml`, triggered on `v*` tag push or manual `workflow_dispatch -f tag=vX.Y.Z`)
- tag-push event 는 GitHub API tag-creation 시 누락되는 케이스가 잦아 (#172) **항상 manual dispatch 로 트리거**: `gh workflow run release.yml -f tag=vX.Y.Z`
- Release Please: Conventional Commits → 자동 version bump + CHANGELOG → PR
- Tag prefix: `v*` (e.g. `v0.1.13`) — `release/v*` is legacy, removed in PR #96
- 수동 편집 금지: version 필드는 Release Please가 관리
- `build-darwin` job runs on `macos-latest` and **does not codesign**. Bun's `--compile` already embeds a linker-signed ad-hoc signature (`Identifier=a.out`, `flags=adhoc,linker-signed`); we ship that as-is. brew/curl install paths don't apply the `com.apple.quarantine` xattr, so Gatekeeper never fires on our distribution channels. Previous re-sign (removed in v0.1.33) only changed `Identifier` and added the Hardened Runtime flag, neither of which affected user-facing behavior. If GUI download distribution is ever added (browser, .dmg), Developer ID Application certificate + notarization (`notarytool submit --wait` + `stapler staple`) will be needed at that point.
- `build-cross` job runs on `ubuntu-latest`, then `apt-get install upx-ucl` + `upx -1 dist/tp-*` to shrink linux binaries (-55% typical). macOS is deliberately **not** UPX-compressed — Gatekeeper/Hardened Runtime SIGKILLs packed Mach-O even with `--force-macos`.
- `release` job signs `checksums.txt` via cosign keyless OIDC + attest-build-provenance, then publishes via `softprops/action-gh-release@v2`.

- **무엇**: `v*` 태그 push 또는 수동 dispatch → `macos-26` 러너에서 앱을 archive+서명+export 후 **App Store Connect(→ TestFlight)** 업로드. ADR-0001 이 제거한 EAS Submit 의 배포 경로를 **Apple 공식 도구만으로**(fastlane/EAS 없이) 복원. 플랫폼별 `archive-and-upload-<plat>` job 이 각자 시크릿 게이트(`guard`/`guard-macos`/…) 로 분리.
- **트리거 (CD primary path)**: 첫 수동 업로드(Task #122) 이후로는 **`release.yml` 의 `testflight` job 이 자동 dispatch** 한다 — `release`+`bump-tap` 성공 후 `gh workflow run testflight.yml --ref <tag> -f tag=<tag>` 로 같은 태그를 **별도 run** 으로 깨운다 (TestFlight 실패가 CLI release 를 red 로 만들지 않게 격리). `GITHUB_TOKEN` 으로 가능 — `workflow_dispatch` 는 recursion 가드의 명시적 예외("workflow_dispatch events always create workflow runs", 공식 docs)라 PAT 불필요, 단 `release.yml` permissions 에 **`actions: write`** 추가 필요. **#172-safe**: 태그를 push event 가 아니라 `-f tag`/`--ref` 로 명시 전달. **load-bearing 제약**: `--ref <tag>` 는 `testflight.yml` 을 *태그 트리* 에서 resolve 하므로(main 아님) — testflight.yml 이 main 에 있는 한 앞으로 자르는 모든 태그는 이를 포함(testflight.yml 존재 *이전* 옛 태그 재릴리즈만 실패; CD 는 현재 main ≤ 태그에만 발화, 첫 업로드는 수동이라 해당 없음). **트리거 = `workflow_dispatch` ONLY (#125)** — testflight.yml 의 옛 `push: tags: [v*]` 직접 트리거는 **제거**됐다. release.yml 도 같은 태그를 dispatch 하므로 둘 다 켜두면 매 릴리즈마다 빌드번호가 다른 **두 run** 이 떠 ASC 에 중복 TestFlight 빌드 + macos-26 archive job 2배 비용이 났다. 이제 release.yml 이 단일 driver, 첫 수동 업로드는 `gh workflow run testflight.yml -f tag=...`.
- **빌드 절반 = 하니스**: `TP_PLATFORM=<plat> scripts/ios.sh archive` (Release archive → `-exportArchive` via 플랫폼별 ExportOptions → `.ipa`/`.pkg`). 빌드 로직은 워크플로에 흩지 않고 하니스에 둠(build/smoke/uitest 와 동일 규율). `cmd_archive` 는 `resolve_archive_params()` 로 ios/ipad/macos/visionos 분기, watchos 만 `die`. **업로드 절반 = 워크플로**: `xcrun altool --upload-app --type <plat>` + ASC API 키(.p8).

ASC 레코드는 iOS 와 **별개**(플랫폼별 레코드, 동일 번들 ID `dev.tpmt.app`/macOS). **[CI run 28328587004 exit-70 수정]** `cmd_archive` 가 export 시 temp ExportOptions 에 `installerSigningCertificate = "3rd Party Mac Developer Installer"` 를 주입한다 — xcodebuild 가 installer identity 를 provisioning profile 이 아닌 keychain 에서 직접 resolve 하게 해 `exportArchive … doesn't include signing certificate` exit 70 을 방지. iOS/visionOS export 는 `ARCHIVE_INSTALLER_CERT` 가 비어 있어 이 주입을 건너뛴다. **non-required.**

- **per-target/per-platform profile 매핑은 `cmd_archive` 가 아니라 `ios/project.yml` 에 박혀야 한다 (런 28318333155 iOS archive `Ad Hoc code signing is not allowed with SDK 'watchOS' / requires a provisioning profile` 수정).** `xcodebuild` 의 CLI build-setting override 는 archive 안 **모든 타깃에 전역 적용**되므로 한 invocation 으로 메인 앱과 임베드 watch 에 서로 다른 `PROVISIONING_PROFILE_SPECIFIER` 를 줄 수 없다. 또 **Manual 서명은 bundle id 로 프로파일을 자동 매칭하지 않는다** (그건 Automatic 서명 동작) — 타깃별 specifier 가 필수다. 그래서 `project.yml` 에 `[config=Release][sdk=…]` 조건부 키로 각 (타깃, 플랫폼) → 프로파일을 매핑한다: 메인 `Teleprompter` 타깃 = `[sdk=iphoneos*]`→`Teleprompter iOS App Store`, `[sdk=macosx*]`→`Teleprompter macOS App Store`, `[sdk=xros*]`→`Teleprompter visionOS App Store`; `TeleprompterWatch` 타깃 `[config=Release]`→`Teleprompter watch app App Store` + `CODE_SIGN_IDENTITY=Apple Distribution`/`CODE_SIGN_STYLE=Manual`. **프로파일 NAME 이 SoT** (`scripts/setup-testflight-secrets.sh` 의 `create_profile` 호출 — iOS :423, visionOS :429, watch :441, macOS :446) — UUID 는 create 마다 랜덤이라 NAME 으로 참조한다. **ad-hoc 키는 `[config=Debug]` 로 격리** (Simulator/smoke/build/test 는 전부 `-configuration Debug` + `$SIGN_FLAGS`; archive 만 `-configuration Release`) — Debug 회귀 없음(iOS smoke 8/8 검증). `DEVELOPMENT_TEAM` 은 archive 시 `TP_DEVELOPMENT_TEAM` 전역 CLI override (전 타깃 동일 팀이라 전역이 맞음).
- **EXPORT 단계도 별도 서명 매핑이 필요하다 — `cmd_archive` 가 export-time 에 temp ExportOptions 로 주입 (런 28323927367 `exportArchive "…app" requires a provisioning profile` exit 70 수정).** `project.yml` 의 `[config=Release][sdk=…]` 키는 **archive** 단계 서명만 고친다 — `xcodebuild -exportArchive` 는 `-exportOptionsPlist` 를 **따로** 읽으므로, `signingStyle=manual` 인 export 는 그 plist 안에 명시적 `provisioningProfiles` dict(+`teamID`)를 요구한다 (manual 은 keychain 의 bundle-id 자동 매칭을 안 한다 — Automatic 서명 동작). 체크인된 `ios/ExportOptions{,.macos,.visionos}.plist` 는 의도적으로 매핑을 비워 둬(team id/프로파일 이름 git 유출 방지) 그대로 쓰면 export 가 깨졌다. `cmd_archive` 가 `resolve_archive_params()` 의 `ARCHIVE_PROFILE_MAP`(bundleId|profileName 쌍, bash-3.2-safe 평행배열)과 `$TP_DEVELOPMENT_TEAM` 으로 `$EXPORT_DIR/ExportOptions.resolved.plist`(임시, `rm -rf` 된 export dir 안 — git 무유출)를 PlistBuddy 로 생성해 export 한다. **iOS 매핑은 2개**(메인 `dev.tpmt.app`→`Teleprompter iOS App Store` + 임베드 watch `dev.tpmt.app.watchkitapp`→`Teleprompter watch app App Store` — `.ipa` 가 두 앱을 담으므로), **macOS/visionOS 는 1개**(watch embed 가 `destinationFilters:[iOS]` 라 그 슬라이스엔 watch 없음). 프로파일 NAME 은 위와 동일 SoT. (잔여 리스크: export 가 `signingCertificate` 까지 요구하면 temp plist 에 `Apple Distribution` 추가 — 현재 로그는 profile 만 요구해 미선반영.)
- **archive 경로의 STDOUT 은 아티팩트 경로 한 줄만 — 빌드 도구의 stdout 누수가 `$GITHUB_OUTPUT` 을 깬다 (런 28326191287 `Unable to process file command 'output'` / `Invalid format '⚙️  Generating project...'` 수정).** 워크플로는 `IPA="$(TP_PLATFORM=ios scripts/ios.sh archive)"` 로 `cmd_archive` 의 stdout 을 캡처해 `echo "ipa=$IPA" >> "$GITHUB_OUTPUT"` 하므로, `cmd_archive` 와 그 호출 사슬(`ensure_xcframework`→`cmd_rust`→`build-xcframework.sh`, `ensure_project`→`cmd_gen`) 전체가 stdout 에 **오직 최종 `.ipa`/`.pkg` 경로** 만 남겨야 한다 — 그 외 한 바이트라도 새면 `$GITHUB_OUTPUT` 의 `key=value` 형식이 깨져 archive+export 가 성공해도(=`.ipa` 생성) job 이 output 캡처 단계에서 fail 한다. 누수원 2곳을 stderr 로 리다이렉트: (1) `cmd_gen` 의 `xcodegen generate` 가 stdout 으로 찍는 `⚙️  Generating project...`(`scripts/ios.sh` `cmd_gen` → `( … xcodegen generate ) >&2`), (2) `build-xcframework.sh gen_bindings` 의 `cargo run --bin uniffi-bindgen`(=`cargo run` 이 자식 바이너리 stdout 을 forward — `( … cargo run … ) >&2`). `cargo build` 와 `xcodebuild`(`| xcbeautify_or_cat >&2`)·`-create-xcframework`(`>&2`)·`log()`(`>&2`)·`die()`(`>&2`)·`PlistBuddy`(파일/`>/dev/null`) 는 이미 stdout-clean. **검증**: `out="$(TP_FORCE_RUST=1 scripts/ios.sh rust 2>/dev/null)"; [ -z "$out" ]` 및 `out="$(scripts/ios.sh gen 2>/dev/null)"; [ -z "$out" ]` — 둘 다 0바이트여야 한다.
- **빌드 번호**: `TP_BUILD_NUMBER=$(( RUN_NUMBER * 100 + RUN_ATTEMPT ))` → `cmd_archive` 가 `CURRENT_PROJECT_VERSION` 오버라이드(ASC 는 재사용 빌드번호 거부, 단조 증가 필수). `run_number*100+run_attempt` 는 run 마다 단조 증가 + "Re-run failed jobs" attempt 마다 유니크 → 업로드 성공 후 재실행이 ASC 의 이미-사용된 CFBundleVersion 과 충돌 안 함. **산술은 반드시 셸 `$(( ))` 에서**
- **`CFBundleVersion`/`CFBundleShortVersionString` = 반드시 변수 참조 (ASC 중복 버전 거부 방지)**: 메인 `Teleprompter` 타깃은 `GENERATE_INFOPLIST_FILE=YES` 와 함께 **명시적 `INFOPLIST_FILE=Teleprompter-Info.plist`** 도 설정한다 — scene manifest + CFBundleURLTypes 를 담기 위해 필수. XcodeGen 이 이 plist 를 `info: properties:` 블록에서 생성할 때, 변수 참조 없이 냅두면 `CURRENT_PROJECT_VERSION` 의 **리터럴값("1")** 이 `CFBundleVersion` 에 구워진다 — 그러면 archive-time `CURRENT_PROJECT_VERSION=<TP_BUILD_NUMBER>` 빌드세팅 오버라이드가 명시적 plist 값에 의해 덮여 효과가 없다. 결과: 모든 TestFlight 빌드가 CFBundleVersion=1 을 출하 → ASC 가 두 번째 업로드부터 "bundle version must be higher than … '1'" 으로 거부. **(run 28333992407 에서 iOS TestFlight 업로드 실패 — 동일 레이턴트 버그가 macOS/visionOS 다음 업로드도 차단할 예정이었음)**. 수정: `ios/project.yml` 의 `info: properties:` 블록에 `CFBundleVersion: "$(CURRENT_PROJECT_VERSION)"` + `CFBundleShortVersionString: "$(MARKETING_VERSION)"` 를 변수 참조 문자열로 선언 → XcodeGen 이 참조를 plist 에 그대로 기록 → xcodebuild 의 `ProcessInfoPlistFile` 단계에서 빌드세팅 값으로 치환. **`GENERATE_INFOPLIST_FILE=YES`** 를 쓰는 `TeleprompterWatch` 타깃(명시적 `INFOPLIST_FILE` 없음, `info:` 블록 없음)은 이 문제가 없다 — Xcode 가 plist 를 직접 생성하며 항상 빌드세팅 값을 주입한다. 수정 범위는 명시적 INFOPLIST_FILE 이 있는 메인 타깃만. — GitHub Actions expression 에는 산술 연산자(`* + - /`)가 없어 `${{ github.run_number * 100 + github.run_attempt }}` 는 워크플로 **파싱 에러**(`*` unparseable)를 내 모든 브랜치 push 가 0-job red run 을 찍는다. `run_number`/`run_attempt` 는 정수 컨텍스트 조회라 env 로 주입 후 `$(( ))` 에 안전히 전개. 버전 문자열은 `project.yml MARKETING_VERSION`.

ASC 레코드 2 → 1(`dev.tpmt.app` 단일 레코드가 watch 슬라이스 포함). **iOS `archive-and-upload` 잡이 watch 빌드를 흡수**: (a) nightly Rust + `rust-src` 설치(arm64_32 watchOS `-Z build-std` 슬라이스 빌드에 필요), (b) `xcodebuild -downloadPlatform watchOS`(watch SDK 확보; Apple SDK CDN 의 간헐 exit 70 flake 에 대비해 3회 백오프 재시도 — run 28578680851 attempt 1 이 여기서 죽고 rerun 은 13초에 통과), (c) iOS 앱 profile + **임베드 watch 앱 profile(`IOS_WATCH_PROVISIONING_PROFILE_BASE64`)** 둘 다 keychain 에 설치. 인증서는 iOS Distribution 재사용, `altool --type ios`(watch 슬라이스는 서버가 watchOS App Store 로 라우팅). (d) **`TP_FORCE_RUST: "1"` 을 archive step env 에 설정 (#125, BLOCKER 수정)** — `TpCore.xcframework` 는 `rust/target/` 아래(ios.sh:114)에 있고 `Swatinem/rust-cache(workspaces: rust)` 가 그 디렉터리를 캐시한다. ci.yml 의 swift-build/swift-smoke 잡이 `TP_SKIP_ARM64_32=1` 로 빌드한 **6-슬라이스(watch device arm64-only, arm64_32 없음)** xcframework 가 캐시에 저장되는데, 그게 복원되면 `ensure_xcframework()` 가 Info.plist 존재만 보고(슬라이스 완전성 미검사, ios.sh:401/411) **재빌드를 스킵** → 임베드 watch 가 필요한 iOS **device** archive 가 `missing architecture(s) ... arm64_32` 로 **runner 프로비저닝+키체인 셋업 후** hard-fail(유료 분 낭비). `TP_FORCE_RUST=1` 이 7-슬라이스(arm64_32 nightly `-Z build-std` 포함) 를 강제 재빌드해 poisoned cache 를 무시한다. 인증서는 iOS Distribution 재사용, `altool --type ios`(watch 슬라이스는 서버가 watchOS App Store 로 라우팅). **non-required.** `ios/ExportOptions.watchos.plist` 삭제됨.
- **macOS/visionOS archive 잡은 `TP_SKIP_ARM64_32: "1"` 을 archive step env 에 설정해야 한다 (런 28318333155 에서 둘 다 `nightly rust-src missing` 으로 fail 한 버그 수정).** 이 두 잡은 watch 를 임베드하지 않으므로 archive 에 macos-fat/xros 슬라이스만 쓰지만, `build-xcframework.sh` 는 `TP_PLATFORM` 과 무관하게 **항상 9 슬라이스 전부**를 빌드하려 한다 — 그 중 watchOS-device 의 arm64_32 sub-슬라이스는 Rust tier-3 라 nightly toolchain + `rust-src`(`-Z build-std`)가 필요하다. iOS 잡은 watch 를 임베드하므로 nightly+rust-src 를 설치하지만 (위 (a)), macOS/visionOS 잡은 stable 1.96.0 만 설치하므로 `ensure_nightly()` 가 `nightly rust-src missing` 으로 die 한다 (`build-xcframework.sh:97-99`). `TP_SKIP_ARM64_32=1` 이 `build_watchos_arm64_32()` 의 nightly 경로를 건너뛰게 한다 (`build-xcframework.sh:131-135`) — 이 한 sub-슬라이스만 빠진 xcframework 는 macOS/visionOS archive 에 전혀 영향이 없다 (watch-device 슬라이스를 안 쓰므로). iOS 잡엔 절대 설정하지 않는다 (arm64_32 가 임베드 watch device archive 에 필수).
- **non-required** (신규 job 규율 — 안정 green 후 승격). 플랫폼별 job(iOS/macOS/visionOS) 전부 non-required 로 진입 (watchOS 는 iOS 잡에 통합, 별도 job 없음). **5플랫폼 TestFlight 스캐폴딩 완성** — 시크릿/ASC 레코드가 채워지면 release 마다 `release.yml` 의 `testflight` job 이 자동 dispatch 해 전 플랫폼 업로드(위 "트리거 (CD primary path)"). 첫 업로드는 수동(`gh workflow run testflight.yml -f tag=...`, Task #122) — Apple 계정 필요.

## Relay Deploy (`deploy-relay.yml`, ADR-0003 Step 8b — Rust 바이너리)
- **빌드**: `ubuntu-latest` 에서 `dtolnay/rust-toolchain@1.96.0`+`Swatinem/rust-cache` 로 `cargo build --release --target x86_64-unknown-linux-gnu --bin tp-relay`(x86_64 Vultr = native build). `TP_BUILD_SHA=${{ github.sha }}` 를 **build env** 로 주입 → `build.rs` 가 읽어 `/health.buildSha` 로 verbatim 노출(full 40-char).
- **배포**: SCP `/tmp/tp-relay` → 호스트, base unit `ExecStart=/usr/local/bin/tp-relay`+`RELAY_PORT=7090` 설치, `systemctl restart tp-relay`. **on-disk `sha256(/usr/local/bin/tp-relay)` 검증** + `/health.buildSha==github.sha` assert(`curl --retry`).
- **시크릿은 base unit 이 아니라 drop-in `/etc/systemd/system/tp-relay.service.d/secrets.conf`**(`TP_RELAY_RESUME_SECRET`/`TP_RELAY_PUSH_SEAL_SECRET[_PREV]`/APNs) — deploy 가 base unit 을 rewrite 해도 systemd 가 drop-in 을 merge 하므로 시크릿 안 지워짐. deploy 워크플로우는 시크릿을 읽지/쓰지 않는다.
- **트리거**: `rust/tp-relay,tp-proto,tp-core`, `rust/Cargo.lock`, `deploy-relay.yml` 자기자신 (구 `packages/relay,protocol,daemon` 제거 — TS relay 퇴역). **flip-live-on-merge** = 머지가 곧 `relay.tpmt.dev` 자동 cutover(downtime-OK, Amendment 1). 시크릿 reissue 는 머지 *전에* 호스트에서.
- Health check: `https://relay.tpmt.dev/health` · Port: 7090 · `timeout-minutes: 30`
- **arm64 호스트 전환 시 주의**: `ubuntu-latest`(x86_64)에서 `aarch64-unknown-linux-gnu` 링크는 `gcc-aarch64-linux-gnu` 크로스 링커가 필요(현 워크플로우 미설치) — 현 Vultr 가 x86_64 라 dead path 지만 전환 시 `cross`/링커 추가 필요.

## Scripts
- `scripts/build.ts`: multi-platform `bun build --compile --minify` (darwin/linux × arm64/x64). Always passes `--minify`; `--bytecode` is deliberately off (+9 MB for -20 ms warm start is a bad trade; download size dominates install UX). Native Windows is unsupported — Windows users run the Linux build under WSL.
- `scripts/install.sh`: curl-pipe-sh installer (macOS/Linux; Windows users run inside WSL)
- `scripts/deploy-relay.sh`: SSH 배포 (arch 자동 감지)
